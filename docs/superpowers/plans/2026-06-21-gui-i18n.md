# GUI i18n + Tab-Completion Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Make every visible GUI string customisable via `messages.yml` (single language, English defaults so nothing changes visually) and give every command sensible tab-completion.

**Architecture:** Add `Messages.list(key,…)` for multi-line lore; move all GUI button/lore text into `gui.*` MiniMessage keys (defaults bundled, auto-merged by the existing `mergeMessagesDefaults()`); GUIs read text from `messages()` instead of literals. A shared `Completions` util backs `onTabComplete` for all commands.

**Tech Stack:** Java 21, Paper API, Adventure/MiniMessage, JUnit 5, MockBukkit.

## Global Constraints

- English defaults must reproduce the CURRENT text verbatim — existing tests assert display-name substrings and must stay green without edits.
- `Messages.list` and all `Completions` methods are fail-soft: never throw into a command/GUI; missing key → empty list.
- Key schema: `gui.<screen>.<element>` and `gui.<screen>.<element>.lore` (lore = a YAML string list). Titles keep their existing `gui-*-title` keys.
- Bundled defaults live in `src/main/resources/messages.yml`; do not remove or rename existing keys.
- Do NOT `git add -A` (gitignored `.claude/` + `.superpowers/` scratch) — stage explicit paths. Run `./gradlew test` before each commit.

```java
/** Reads a YAML string-list at {@code key} as multi-line lore: one Component per line, italic off. */
public java.util.List<net.kyori.adventure.text.Component> list(String key, String... placeholders) {
    java.util.List<String> lines = config.getStringList(key);
    java.util.List<net.kyori.adventure.text.Component> out = new java.util.ArrayList<>(lines.size());
    for (String line : lines) {
        out.add(deserialize(line, placeholders)
            .decoration(net.kyori.adventure.text.format.TextDecoration.ITALIC, false));
    }
    return out;
}
```

```java
package de.derfakegamer.sentinel.util;

import org.bukkit.Bukkit;
import org.bukkit.entity.Player;
import java.util.Collection;
import java.util.List;

/** Case-insensitive, sorted prefix filters for command tab-completion. All fail-soft. */
public final class Completions {
    private Completions() {}
    private static final List<String> DURATIONS = List.of("1h", "6h", "12h", "1d", "7d", "30d", "perm");

    public static List<String> filter(String prefix, Collection<String> options) {
        String p = prefix == null ? "" : prefix.toLowerCase();
        return options.stream().filter(o -> o != null && o.toLowerCase().startsWith(p)).sorted().toList();
    }
    public static List<String> of(String prefix, String... options) { return filter(prefix, List.of(options)); }
    public static List<String> players(String prefix) {
        return filter(prefix, Bukkit.getOnlinePlayers().stream().map(Player::getName).toList());
    }
    public static List<String> durations(String prefix) { return filter(prefix, DURATIONS); }
    public static List<String> reasons(String prefix, List<String> presets) { return filter(prefix, presets); }
}
```

## GUI i18n batches (Tasks 4–6)

**Shared procedure for every GUI in a batch:**

1. For each hardcoded `Component.text("…", COLOR)` button NAME, add a `gui.<screen>.<element>` key to
   `src/main/resources/messages.yml` whose value reproduces it in MiniMessage (e.g. `<red>Ban`,
   `<aqua>Player Manager`). Replace the literal with `plugin.messages().plain("gui.<screen>.<element>")`.
2. For each lore list, add a `gui.<screen>.<element>.lore` YAML string list (one entry per line,
   MiniMessage, dynamic parts as `<placeholder>`), and replace the literal lore with
   `plugin.messages().list("gui.<screen>.<element>.lore", "<name>", value, …)`.
3. Status-dependent labels get one key per state (`…ban` / `…unban`).
4. The English default MUST equal the current rendered text verbatim.
5. Update the matching `*Test` only if it asserted a hardcoded LORE string; display-name substring
   assertions stay green because defaults are unchanged.

**Worked example — converting one button (PlayerActionsGui "Ban"):**

Before:
```java
inventory.setItem(BAN, Items.button(Material.BARRIER,
    Component.text(banned ? "Unban" : "Ban", banned ? NamedTextColor.GREEN : NamedTextColor.RED),
    List.of(hint(banned ? "Remove this player's ban" : "Permanently ban this player"))));
```
messages.yml:
```yaml
gui:
  actions:
    ban: "<red>Ban"
    unban: "<green>Unban"
    ban-lore: ["<gray>Permanently ban this player"]
    unban-lore: ["<gray>Remove this player's ban"]
```
After:
```java
inventory.setItem(BAN, Items.button(Material.BARRIER,
    plugin.messages().plain(banned ? "gui.actions.unban" : "gui.actions.ban"),
    plugin.messages().list(banned ? "gui.actions.unban-lore" : "gui.actions.ban-lore")));
```

After converting a batch: run that batch's GUI tests + `./gradlew compileJava compileTestJava`, then the full suite.
